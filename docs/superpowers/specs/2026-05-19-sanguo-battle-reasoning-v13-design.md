# 三國演義探險地圖 v13：戰役推理設計規格

## 1. 設計目標

`戰役現場` 在 v12 中主要是事件清單與局部關係圖，能看見參與者，但學習目標不夠清楚。v13 要把它改成 `戰役推理`：讓國小 3-6 年級學生用一場戰役理解故事因果，並透過 GraphRAG 的節點與關係證據完成簡短判斷。

核心學習句：

> 先看一場戰役為什麼發生，再找誰做了關鍵決定，接著看行動如何改變結果。

## 2. 目標使用者

- 年齡：國小 3-6 年級。
- 使用情境：手機或平板上自主探索，也可由老師投影帶讀。
- 已知程度：可能知道劉備、曹操、孫權等主角，但不熟悉章回順序與戰役因果。
- 主要困難：容易被太多人物與線條分散注意，難以知道「這場戰役到底重要在哪裡」。

## 3. 設計原則

1. `一場戰役一次只教一件核心轉折`
   每場戰役不追求完整百科，而是聚焦一個最適合孩子理解的故事問題。

2. `先因果，後關係`
   不先列人物，也不先展示一堆關係線。先讓孩子知道故事骨架，再看人物如何造成變化。

3. `任務是閱讀引導，不是裝飾文字`
   每個任務都要讓孩子做一個具體動作：找原因、找轉折、說判斷。

4. `GraphRAG 證據要可回到節點與關係`
   每張證據卡都要能連到地圖上的人物、事件、地點或關係線，避免只剩文字摘要。

5. `手機優先`
   戰役推理在手機上要避免左右分欄與多重捲動，主要流程應該是「選戰役 -> 看四格 -> 看地圖 -> 看證據」。

## 4. 模式命名與入口

原本：

- Tab 名稱：`戰役現場`
- 描述：`選事件看參與者`

v13 建議：

- Tab 名稱：`戰役推理`
- 描述：`看原因與轉折`

進入後不顯示一長串事件。預設顯示三個推薦戰役卡：

- `官渡之戰`：以少勝多
- `赤壁之戰`：聯盟與火攻
- `長坂坡`：撤退與保護

如果章節範圍不含上述戰役，則依目前章節資料挑出權重最高的 3-5 個事件，並以同樣格式呈現。

## 5. 戰役四格推理模型

每場戰役用四格組織：

1. `起因`
   回答：這場戰役為什麼會發生？

2. `決策`
   回答：誰做了關鍵選擇？

3. `行動`
   回答：用了什麼計策、合作、衝突或撤退？

4. `結果`
   回答：誰得利？誰受損？局勢怎麼變？

四格不是靜態海報，而是互動篩選器。點某一格時：

- 地圖只高亮該格相關節點與關係。
- 右側證據卡只顯示該格相關證據。
- 地圖標題改成 `赤壁之戰：行動` 這種可理解格式。

## 6. 推薦戰役內容框架

### 6.1 官渡之戰

- 核心問題：曹操為什麼能以少勝多？
- 起因：袁紹與曹操爭奪北方主導權。
- 決策：曹操採納關鍵情報與奇襲策略。
- 行動：攻擊烏巢、打擊袁紹糧草。
- 結果：袁紹勢力衰退，曹操成為北方強權。

### 6.2 赤壁之戰

- 核心問題：弱勢的一方如何靠聯盟與計策扭轉局勢？
- 起因：曹操南下，江東必須決定是否抵抗。
- 決策：孫權、周瑜、諸葛亮推動孫劉合作。
- 行動：黃蓋詐降、火攻、連環計等線索形成轉折。
- 結果：曹操敗退，孫劉暫時合作，三方格局更明顯。

### 6.3 長坂坡

- 核心問題：撤退時，什麼比勝利更重要？
- 起因：曹操追擊劉備，劉備一方陷入危急。
- 決策：劉備需要保住人心與隊伍，趙雲、張飛承擔保護任務。
- 行動：趙雲救阿斗，張飛斷橋阻敵。
- 結果：劉備雖敗退，但保住重要人物與凝聚力。

## 7. 任務式探索

每場戰役固定三個任務：

1. `找原因`
   從四格中的 `起因` 找出一個人物或勢力，說明戰役為什麼發生。

2. `找轉折`
   從 `決策` 或 `行動` 找出一條關係或一張證據卡，說明局勢怎麼改變。

3. `說判斷`
   用一句話回答：如果你是其中一位人物，你會做同樣的選擇嗎？為什麼？

任務不需要複雜打分。完成方式可以是點選對應四格與一張證據卡後，顯示 `已找到證據` 的狀態。`說判斷` 可連到既有 `想一想` 右側抽屜或簡短輸入區。

## 8. 版面設計

### 8.1 桌機 / 平板橫向

- 左側：戰役選擇與四格推理。
- 中央：D3 關係地圖。
- 右側：戰役證據卡。
- 底部或右側保留重要時刻，但戰役模式中要弱化，不搶主流程。

### 8.2 手機

手機不使用三欄。建議順序：

1. 上方模式列：`人物探險`、`戰役推理`、`關係路徑`。
2. 戰役卡橫向或 2 欄排列。
3. 四格推理作為 sticky segmented control。
4. 地圖區。
5. 下方證據抽屜，可展開閱讀。

避免：

- 左側抽屜與右側抽屜同時存在。
- 篩選區太高導致地圖被推到折線以下。
- 使用者點戰役後不知道下一步。

## 9. 地圖呈現

每場戰役建議顯示 8-15 個主要節點：

- 戰役 / 事件節點：菱形或圓角方塊。
- 人物節點：圓形，保留陣營色。
- 地點節點：較淡色方塊。
- 關鍵轉折關係：較粗線。
- 非當前四格的節點：降低透明度，不完全移除。

點四格時，圖上要有明確差異：

- `起因`：高亮勢力、地點、背景事件。
- `決策`：高亮主帥、謀士、命令或說服關係。
- `行動`：高亮計策、攻擊、援助、撤退關係。
- `結果`：高亮勝負、佔領、逃亡、勢力變化關係。

## 10. 戰役證據卡

原本右側 `推薦故事與戰役` 要改成 `戰役證據`。

每張卡格式：

- 小標：`起因`、`決策`、`行動`、`結果`。
- 標題：人物或事件名稱。
- 內容：一段不超過 40 字的白話說明。
- 來源：第幾回。
- 動作：`在地圖上看`。

範例：

```text
行動
黃蓋詐降
黃蓋假裝投降，讓曹操放鬆戒心，火攻才有機會成功。
第 47-49 回
[在地圖上看]
```

## 11. 資料與演算法

### 11.1 最小可行資料

先不要求 LLM 重新抽圖。v13 可用現有 GraphRAG 資料做第一版：

- 使用 `node.type === "battle" || node.type === "event"` 找候選戰役。
- 用 `chapters` 判斷章回範圍。
- 用關係 `kidCategory` 粗分四格：
  - 起因：`story`、`place`、`office`
  - 決策：`command`、`strategy`
  - 行動：`strategy`、`enemy`、`ally`
  - 結果：`defeat`、`enemy`、`story`

### 11.2 推薦戰役人工覆寫

為了讓教學品質穩定，三個代表戰役應有少量人工設定：

```js
const BATTLE_LESSON_PRESETS = {
  "赤壁": {
    title: "赤壁之戰",
    theme: "聯盟與火攻",
    coreQuestion: "弱勢的一方如何靠聯盟與計策扭轉局勢？",
    steps: [
      { id: "cause", label: "起因", focusCategories: ["story", "place"] },
      { id: "decision", label: "決策", focusCategories: ["command", "strategy"] },
      { id: "action", label: "行動", focusCategories: ["strategy", "ally", "enemy"] },
      { id: "result", label: "結果", focusCategories: ["defeat", "story"] }
    ]
  }
}
```

人工設定只負責教學框架，不取代 GraphRAG 證據。證據仍從現有節點與關係中挑選。

## 12. 狀態與 URL

戰役模式應支援分享：

- `mode=battle`
- `battle=赤壁之戰` 或 `event=赤壁`
- `step=cause|decision|action|result`

使用者打開分享連結時，應直接進入該戰役與該四格。

## 13. 成功標準

### 13.1 學習成功

- 孩子能說出一場戰役的 `起因 -> 決策 -> 行動 -> 結果`。
- 孩子能指出至少一個人物如何改變局勢。
- 孩子能用一張證據卡支持自己的判斷。

### 13.2 UX 成功

- 進入 `戰役推理` 後，不需要讀說明也知道要先選一場戰役。
- 點選戰役後，畫面立刻顯示四格流程。
- 點四格後，地圖和證據卡同步變化。
- 手機上不需要在多個抽屜間來回跳動。

### 13.3 技術成功

- 不破壞 `人物探險` 與 `關係路徑`。
- 單檔 HTML 仍可分享。
- 預設節點控制在 8-15 個，最多不超過 20 個。
- 使用現有 v12 GraphRAG payload，不新增外部 API 依賴。

## 14. 非目標

- 不做完整戰役百科。
- 不做複雜計分或登入成就系統。
- 不要求每一場事件都人工整理成四格。
- 不在 v13 引入後端、Neo4j 或 API。
